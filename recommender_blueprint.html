🧠 DLRM Model Class models/dlrm.py 
class DLRM(nn.Module):
    def __init__(self, num_users, num_items,
                 embed_dim=128, dense_in=2):
        self.user_emb = nn.Embedding(num_users, embed_dim)
        self.item_emb = nn.Embedding(num_items, embed_dim)
        self.dense_bottom = nn.Linear(dense_in, embed_dim)
        self.mlp = nn.Sequential(
            nn.Linear(384, 64), nn.ReLU(),
            nn.Linear(64, 32),  nn.ReLU(),
            nn.Linear(32, 1),   nn.Sigmoid()
        )

Per-User Recommendations

New /recommend/{user_id} endpoint: Instead of predicting a single user+item pair and matching against TMDB popularity, this endpoint does REAL personalization. Given a user_id, it runs ONE forward pass through the DLRM for ALL 1,682 items in the catalogue. Each item gets a predicted score between 0 and 1. The top-K highest scores (default 10) are returned, each enriched with movie metadata (title, genres, year). No TMDB popularity involved — the model itself decides the ranking. This is the difference between “here are popular movies near your vibe” and “here are the specific movies the model thinks YOU will love.” Cold-start fallback: If the user is unknown, the endpoint skips scoring and returns pre-computed popular items with a "note": "cold-start: popularity-based fallback" field. Only returns 404 if no popular items are available.

🛡 The Embedding Vault

Where 93% of the model's knowledge lives — learned identity fingerprints for every user and every movie.

⚠️ The Problem: IDs Are Meaningless

User 42 and Movie 567 are just numbers. Numbers tell you nothing about preferences. You need a way to capture WHO someone is and WHAT something is — not just their ID number.

Embeddings solve this: they replace every bare ID with a learned fingerprint — a list of 128 numbers that captures taste, genre affinity, mood, and more. The model discovers these patterns on its own during training.

What Happens in Each Room?

The Big Room (384 → 64) — 24,640 connections

Every ingredient in the mega-sandwich gets examined. The room has 64 helpers, each looking at ALL 384 ingredients. They figure out: which combinations matter?

Then ReLU: Anything negative gets thrown out. Only positive signals survive.

The Medium Room (64 → 32) — 2,080 connections

The 64 surviving signals get examined again. Now only 32 helpers, looking for higher-order patterns.

ReLU again — negatives get tossed.

The Tiny Room (32 → 1) — 33 connections

32 signals compressed to 1. Then Sigmoid squishes it between 0 and 1.

🍳 The Prep Kitchen

Before any prediction happens, raw MovieLens data needs to be transformed. This shared module ensures training and serving use the exact same transforms — preventing train/serve skew.

📁 data/preprocessing.py

A single Python module with 6 functions that handle everything from loading raw tab-separated data to producing train/test tensors. Every line of feature engineering lives here — nowhere else.

The Six Functions

load_movielens_data (data_path) → np.ndarray

Loads raw u.data from disk using np.loadtxt. Returns a (N, 4) int64 array of [user_id, item_id, rating, timestamp].

build_id_mappings (raw) → (user2idx, item2idx)

Maps raw user/item IDs (sparse, arbitrary numbers) to contiguous indices starting at 0. These indices become the lookup keys for the embedding tables.

compute_user_stats (train_raw) → (user_ratings, max_count)

Computes per-user rating lists and the maximum rating count — from training data only. This is the critical function for preventing data leakage. Test features use these training-derived statistics.

make_features (data, user2idx, item2idx, user_ratings_dict, max_count) → (cont, cat, targets)

Builds three arrays: continuous features (normalised mean rating & interaction count), categorical indices (user & item IDs for embedding lookup), and normalised targets (rating / 5.0). Unseen users default to a mean of 3/5.

prepare_splits (raw) → 9-tuple

The orchestrator. Sorts by timestamp, performs an 80/20 split, builds ID mappings from all data, computes stats from training data only, then calls make_features for both splits. Returns everything the training script needs.

load_item_metadata (data_dir) → dict

Loads item metadata (titles, genres, release dates) from the u.item file. Returns a dictionary keyed by item ID with title, genre, and year information — used by the API to enrich recommendations with human-readable details.

⚠️ Key Design Decision: No Data Leakage

compute_user_stats runs on TRAINING data only. Test features use training statistics — this prevents data leakage. If test stats leaked into training features, the model would appear better than it actually is, and real-world performance would be worse.

💡 One Source of Truth

Both train_movielens.py and api/app.py import from this module — same transforms, one source of truth. If you change how features are built, both training and serving update together. No copy-paste drift. No silent bugs where training normalises one way and serving normalises another.

The Pipeline Flow

load_movielens_data u.data → (N, 4) array
↓
prepare_splits sort by timestamp → 80/20 split
↓
build_id_mappings all data
compute_user_stats train only
↓

↓
Ready for Training cont + cat + targets for both splits
🌟 Cookie Monster’s Prep Kitchen

Before the Talent Show can start, Cookie Monster runs the Prep Kitchen. Raw ingredients arrive — messy bags of flour, sugar, eggs, all labeled with weird codes.

First, Cookie Monster LOADS all the bags onto the counter. Then he gives every ingredient a CLEAN NUMBER — Flour is 0, Sugar is 1, Eggs are 2. No more confusing codes.

Then comes the important rule: Cookie Monster tastes ONLY the practice batch to figure out the recipe. He never peeks at the contest batch! If he tasted the contest cookies before judging them, that would be CHEATING.

And the best part? Cookie Monster wrote his recipe on ONE card. Everyone who needs to bake — whether it’s training day or show day — uses the same card. No one copies it by hand. No one makes their own version. One recipe, everywhere.

An A/B testing framework that trains two models on identical data, collects per-user metrics, and runs rigorous statistical analysis to determine if one model truly outperforms the other.

📁 scripts/experiment_framework.py

Two dataclasses (ExperimentConfig and ExperimentResult), one runner class (ExperimentRunner), three statistical helper functions, and a report generator. Everything you need to run, analyse, and document a model comparison.

The Building Blocks

ExperimentConfig dataclass

Defines every parameter for both models in a single frozen blueprint. Model A (control) and Model B (treatment) each get their own embedding_sizes, mlp_layers, lr, and epochs. Shared settings include primary_metric (default NDCG@10), significance_level (0.05), and random_seed (42).

ExperimentResult dataclass

Holds everything produced by a completed experiment: aggregate metrics per model, per-user metric arrays (the raw material for statistical tests), training times, and all computed statistics — t-statistic, p-value, confidence interval, Cohen’s d, power, significance flag, and winner. Includes a to_dict() method for JSON serialisation.

ExperimentRunner class

The engine. Takes an ExperimentConfig, loads data once, trains both Model A and Model B on the exact same training set, evaluates both on the same test set, collects per-user NDCG/Precision/Recall/HitRate, then runs the full statistical battery. Two modes: run() for standard A/B and run_interleaved() for interleaved experiments.

Per-User Metric Collection

Aggregate metrics (mean NDCG = 0.73) hide variance. Two models might have the same average but wildly different per-user distributions. The framework collects per-user metric arrays — one NDCG score, one Precision@K, one Recall@K, and one HitRate@K for every user who has at least 2 test interactions and at least 1 relevant item. These arrays become the input to every statistical test.

model_a_per_user["NDCG@10"] = [0.81, 0.65, 0.92, 0.43, ...]
model_b_per_user["NDCG@10"] = [0.84, 0.71, 0.88, 0.50, ...]
# One value per user → paired statistical tests

The Statistical Battery

Welch’s t-test scipy.stats.ttest_ind(equal_var=False)

Tests whether the mean difference between Model A and Model B per-user metrics is statistically significant. Uses Welch’s variant (unequal variance assumed) — safer than Student’s t-test because the two models may produce metric distributions with different spreads. Returns a t-statistic and p-value.

95% Confidence Interval compute_confidence_interval(a, b)

Computes the confidence interval for the mean difference (B − A). Uses the pooled standard error and a t-critical value at the configured significance level. If the interval excludes zero, the difference is significant. More informative than a binary pass/fail — it tells you how much better one model might be.

Cohen’s d Effect Size compute_cohens_d(a, b)

Measures practical significance. A p-value can be tiny with a huge sample but the actual difference negligible. Cohen’s d normalises the mean difference by the pooled standard deviation. Interpretation: |d| < 0.2 = negligible, < 0.5 = small, < 0.8 = medium, ≥ 0.8 = large.

Statistical Power compute_power(effect_size, n, alpha)

The probability of detecting a real difference when one exists. Uses the non-central t-distribution approximation. If power < 0.8, the experiment may not have enough users to draw reliable conclusions — a “not significant” result might just mean “not enough data.” The report warns when power is low.

Interleaved Experiments

🔀 run_interleaved()

Standard A/B tests compare separate user populations. Interleaved experiments remove population bias: both models score every user, but each user is randomly assigned to either group A or B. This controls for temporal shifts and user-population differences — the only variable is the model itself.

assignments = np.random.choice(["A", "B"], size=n_users)
# Each user randomly routed to A or B group
# Same model trains, different attribution

Markdown Report Generation

📄 to_report(result)

Generates a complete Markdown report from an ExperimentResult. Includes model comparison tables, per-metric deltas, the full statistical analysis table, and an automated interpretation section that classifies the effect size (small/medium/large) and recommends the winner for deployment. Warns when statistical power is below 0.8.

⚠️ Key Design Decision: Same Data, Fair Fight

Both models train on the exact same data split with the same random seed. The only variables are architecture, learning rate, and epochs. Without this control, a difference in metrics could be caused by different training data rather than the model itself.

⚠️ Why Welch’s, Not Student’s?

Student’s t-test assumes both groups have equal variance. But a deeper model might produce a tighter distribution of NDCG scores (less variance) than a shallow baseline. Welch’s t-test relaxes this assumption — it adjusts degrees of freedom based on each group’s actual variance. Always the safer choice.

🎯 Classical Baselines

Before the DLRM can claim the throne, it has to prove it beats the old guard. These hand-crafted feature models are the yardstick — if the neural network can’t outperform a gradient-boosted tree on 8 features, something is wrong.

📁 models/classical.py

Two classes that give classical ML models the same ranking interface as DLRM. FeatureBuilder computes 8 hand-crafted features per user-item pair. ClassicalRanker wraps any sklearn classifier so compute_metrics can evaluate them identically to the deep model.

FeatureBuilder — The 8 Features

👤 User Stats (features 0–3)
user_mean_rating

Average rating the user gave, normalised to [0, 1] by dividing by 5. Defaults to 3/5 for cold-start users.

user_rating_count

How many ratings the user gave, normalised by the max count across all users. Captures activity level.

user_rating_var

Variance of the user’s ratings. A high value means the user is polarised — loves some, hates others. Low means a flat rater.

user_days_active

Days between first and last training rating, normalised. Measures engagement span.

🎬 Item Stats (features 4–6)
item_mean_rating

Average rating the item received, normalised. A proxy for overall quality.

item_rating_count

How many ratings the item received, normalised by the max. Popularity signal.

item_popularity_rank

Rank-ordered popularity normalised to [0, 1]. Most popular item = 1.0.

💡 Deviation Signal (feature 7)
user_mean - item_mean

If a generous rater (mean 4.5) meets a poorly-rated movie (mean 2.0), the deviation is +0.5. This single feature captures how much the user deviates from the crowd for this specific item — the most predictive hand-crafted signal.

ClassicalRanker — The Wrapper

ClassicalRanker (model, feature_builder, name)

Wraps any fitted sklearn classifier that has predict_proba(X). Exposes a rank(user_idx, item_indices) → scores method identical to the DLRM ranker. This lets compute_metrics evaluate logistic regression, random forest, and DLRM with the same code path.

rank() (user_idx, item_indices) → np.array

Calls feature_builder(user_idx, item_indices) to build the (N, 8) feature matrix, then passes it through model.predict_proba. Returns the positive-class probability column as scores. Higher = more likely to be a 4+ rating.

build_matrix() (data_raw) → (X, y)

Convenience method on FeatureBuilder that builds the full (n_samples, 8) feature matrix and binary labels (rating ≥ 4 = positive) for an entire dataset. Used to train the sklearn classifiers before wrapping them in ClassicalRanker.

⚠️ Why Classical Baselines Exist

These are the features a human analyst would compute. The DLRM learns similar patterns automatically via embeddings, but these baselines prove the DLRM is actually adding value. If a logistic regression on 8 hand-crafted features beats the neural network, the neural network is wasting compute. The baselines are the bar the DLRM must clear.

The Baseline Pipeline

train_raw + user2idx + item2idx from preprocessing.py
↓
FeatureBuilder.__init__ pre-compute all user/item stats from training data only
↓
build_matrix(train_raw) → X, y (n_samples, 8) features + binary labels
↓
sklearn.fit(X, y) LogisticRegression / RandomForest / GBT
↓
ClassicalRanker.rank() same interface as DLRM → compute_metrics works for both

🏋 The Training Loop

How the DLRM learns from MovieLens 100K — from raw data loading through evaluation metrics and latency benchmarking. Everything lives in scripts/train_movielens.py.

Phase 1: Data Loading

📁 Preprocessing Module

The training script imports 5 functions from data/preprocessing.py: load_movielens_data, prepare_splits, compute_user_stats, build_id_mappings, plus the constants NUM_FEATURES and RATING_MAX.

prepare_splits(raw) returns a 9-tuple: train continuous features, train categorical features, train targets, test continuous features, test categorical features, test targets, user2idx mapping, item2idx mapping, and the raw test array (needed for per-user evaluation grouping).

Phase 2: MovieLensDataset

MovieLensDataset(Dataset) PyTorch Dataset

A minimal PyTorch Dataset subclass. Takes three NumPy arrays (continuous, categorical, targets) and converts them to tensors in __init__. Continuous features become float32, categorical become long (required for embedding lookups), targets become float32.

__getitem__ returns a tuple: (cont[idx], cat[idx], targets[idx]). That’s it — no transforms, no augmentation. The preprocessing module already did all the work.

Phase 3: DataLoader Setup

batch_size = 256

Each training step sees 256 user-item pairs at once. Small enough to fit in memory, large enough for stable gradient estimates. ~313 batches per epoch for 80K training samples.

shuffle = True

Shuffling prevents the model from learning the order of the data. Without it, the model might memorise batch patterns rather than learning real user-item preferences.

Phase 4: Loss Function & Optimizer

nn.BCELoss() Binary Cross-Entropy

Ratings are normalised to [0, 1] by dividing by RATING_MAX (5). The model’s final layer uses Sigmoid, so outputs are already in [0, 1]. BCELoss measures how far the predicted probability is from the true normalised rating. It penalises confident wrong answers more harshly than uncertain ones.

Adam Optimizer lr = 0.001

A single torch.optim.Adam optimizer handles all parameters — both the dense MLP weights and the sparse embedding tables. Adam adapts the learning rate per-parameter using running estimates of the first and second moments of the gradient. The learning rate of 0.001 is the standard default.

Phase 5: The Training Loop

for epoch in range(1, 21) 20 full passes over the training data
↓
model.train() enable training mode (dropout, batch norm behave differently)
↓
for cont_b, cat_b, target_b in train_loader iterate through ~313 batches of 256 samples
↓
optimizer.zero_grad() clear accumulated gradients from previous step
↓
preds = model(cont_b, cat_b).squeeze() forward pass — continuous features + embedding lookups → MLP → sigmoid
↓
loss = criterion(preds, target_b) BCELoss: how wrong were we?
↓
loss.backward() backward pass — compute gradients for every parameter
↓
optimizer.step() update all weights using Adam’s adaptive learning rates
💡 W&B Logging

After each epoch, the average training loss is logged to Weights & Biases via wandb.log({"train_loss": avg_loss, "epoch": epoch}). This creates live loss curves you can monitor during training.

Phase 6: Evaluation

How Evaluation Works

After training, the model is set to model.eval(). For each user in the test set (minimum 2 interactions, at least 1 relevant item), the model scores all of that user’s test items. Items with rating ≥ 4 (normalised ≥ 0.8) are considered relevant. Scores are ranked descending to produce a top-K list.

NDCG@10

Normalised Discounted Cumulative Gain. Measures ranking quality — relevant items at the top of the list score higher than relevant items buried lower. Normalised against the ideal ordering.

Precision@10

Of the top 10 items recommended, how many were actually relevant? Simple fraction: hits_in_k / k.

Recall@10

Of all relevant items for this user, how many appeared in the top 10? hits_in_k / len(relevant). High recall means we’re finding most of what the user likes.

HitRate@10

Binary: did at least one relevant item appear in the top 10? 1.0 if yes, 0.0 if no. Averaged across all users to get the overall hit rate.

AUC

Area Under the ROC Curve via sklearn.metrics.roc_auc_score. Measures how well the model separates relevant from non-relevant items across all test interactions (not per-user). Only computed when both positive and negative labels exist.

Phase 7: Inference Latency Benchmarking

After evaluation, the script benchmarks single-sample inference speed. A dummy input (1 × NUM_FEATURES continuous + 1 × 2 categorical) is created.

100 warmup runs ensure JIT compilation and cache warming don’t pollute the measurements.

Then 1,000 timed runs using time.perf_counter() (nanosecond precision). Each run’s latency in milliseconds is recorded.

Reported stats: Mean, Median, P95, P99, Min, Max. The P95 and P99 are what matter for production SLAs — they tell you the worst-case latency for 95% and 99% of requests.

Phase 8: Model Saving & Serving Context

torch.save(state_dict)

Saves only the model weights to trained_model_movielens.pth. The architecture is defined in code — you recreate the model structure then load the weights into it.

serving_context.pkl

A pickle file containing everything the API needs at serving time: user2idx, item2idx, idx2item (reverse mapping), user_features (precomputed continuous features per user), and max_count. This bridges training and serving. If the file is corrupted at load time, the app logs the error and continues without it — returning 503 for recommendation requests until a valid context is restored.

⚠️ Key Detail: Training-Only Stats in Serving Context

The serving context’s user_features are computed from raw[:int(len(raw) * 0.8)] — the training portion only. This means the API serves predictions using the same statistics the model was trained on. No leakage at serving time either.

The Complete Pipeline

① Load & split data load_movielens_data → prepare_splits → 9-tuple
↓
② Build model + DataLoader DLRMModel(2, [943,1682], [128,64,32]) + DataLoader(bs=256, shuffle)

🌟 Ernie’s Talent Show Rehearsal

Ernie is training judges for the Talent Show. He has a big binder of past performances — which acts each monster liked, which ones they walked out on.

First, Ernie organises the binder. He sorts everything by DATE and splits it: the first 80% is the PRACTICE BINDER, the last 20% is the SECRET TEST — performances the judges have never seen.

Then Ernie runs rehearsals. He shows the judges 256 performances at a time (a batch). After each batch, the judges guess: “Would this monster like this act?” Ernie checks the answer and says “You were THIS wrong” (that’s the loss). The judges adjust their thinking a tiny bit (that’s the optimizer step).

They do this 20 TIMES through the whole practice binder. By round 20, they’re pretty good at guessing.

Now comes the real test. Ernie pulls out the Secret Test performances. For each monster, he asks the judges to RANK all the acts: “Which ones will this monster love?” Then he checks: Did they put the good acts in the top 10? That’s NDCG. Did most of their top 10 picks actually hit? That’s Precision. Did they find most of the hidden gems? That’s Recall.

Finally, Ernie times the judges: “How fast can you score one act?” He throws away the first 100 answers (they’re still warming up), then times the next 1,000. The 95th percentile time is what matters — that’s the promise to the audience: “Almost all recommendations arrive this fast.”

When the judges are ready, Ernie writes down everything they learned in a notebook (the model weights) and a cheat sheet (the serving context) so anyone running the Talent Show can use the same judges without retraining them.
